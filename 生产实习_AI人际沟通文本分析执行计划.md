# 生产实习：大学生使用生成式 AI 辅助人际沟通行为研究执行计划

## 1. 项目概述

### 1.1 研究题目

从“不会回”到“问 AI”：大学生使用生成式 AI 辅助人际沟通的行为影响因素分析与可视化研究

### 1.2 研究对象

本项目关注的不是一般意义上的“AI 聊天”，而是大学生或年轻群体在现实人际沟通中，把生成式 AI 当作表达辅助、关系判断、情绪表达和社交决策工具的行为。

典型场景包括：

- 不知道怎么回消息，所以问 AI。
- 不知道怎么表达情绪，所以让 AI 生成文案。
- 不知道对方是什么意思，所以让 AI 分析聊天记录。
- 不知道如何处理恋爱、友情、宿舍、师生、社团、实习关系，所以让 AI 提供沟通建议。
- 担心说错话、显得低情商、得罪人，所以让 AI 帮忙润色表达。
- 担心 AI 代聊、AI 味、隐私泄露、表达不真诚，所以产生抵触或边界感。

### 1.3 个人负责内容

本人负责以下模块：

- 网络爬虫与公开文本采集。
- 多平台数据清洗与字段统一。
- 文本分词、词频统计、情感倾向和主题标签分析。
- 场景、动机、风险、态度等多维标签体系落地。
- 数据统计表和可视化图表生成。
- 网络文本分析章节草稿撰写。

### 1.4 最终目标

形成一套从公开文本采集到数据分析、可视化展示和报告写作的完整流程，能够回答以下四个核心问题：

1. 大学生主要在哪些人际沟通场景中求助 AI？
2. 他们为什么会问 AI？
3. 他们如何看待 AI 介入真实人际关系？
4. AI 辅助沟通的主要风险和边界在哪里？

### 2.3 方案选择

本项目采用“稳妥路线 + 部分增强分析”的方案。

稳妥路线：

- 以微博作为主数据源，确保清洗后不少于 900 条有效文本。
- 小红书和知乎作为补充数据源，用于丰富场景案例和观点解释。
- 微信公众号作为背景材料和长文本观点源，不作为核心统计样本。
- 以规则词典进行标签分类，并通过人工抽样复核提高可信度。

采集工具路线：

- 主工具使用 MediaCrawler。它适合作为本项目的统一采集工具，原因是其支持微博、小红书、知乎、B站、贴吧等多平台公开信息采集，并具备 WebUI、关键词搜索、评论抓取和多格式导出能力。
- 小红书设置备用工具 ReaJason/xhs。小红书经常受到登录态、风控策略和页面结构变化影响，如果 MediaCrawler 的小红书模块调不通，则使用 ReaJason/xhs 采集小红书笔记和评论。
- 若小红书自动化采集仍不稳定，则改为半自动采集：人工筛选高相关笔记链接，再用浏览器自动化或手工表格录入标题、正文片段、互动数和评论摘要。
- 微信公众号不作为自动化主采集对象，采用半自动整理，避免把时间消耗在不稳定的公众号爬虫链路上。

评论区分析路线：

- 本项目不把评论区简单当作普通文本混入主样本，而是采用“帖子表 + 评论表”的双层数据结构。
- 帖子表用于回答“原帖提出了什么问题、表达了什么态度、包含什么主题”。
- 评论表用于回答“评论区如何回应原帖，是支持、反对、调侃、风险提醒，还是补充经验”。
- 后续核心分析要保留 `parent_post_id`，确保评论和原帖可以关联，避免评论脱离语境后变成孤立文本。

增强分析：

- 完成场景-风险交叉热力图。
- 完成场景-态度交叉堆叠柱状图。
- 完成平台差异对比图。
- 时间充足时补充动机-使用方式桑基图和高频词共现网络图。

## 4. 数据采集计划

### 4.1 平台优先级

| 优先级 | 平台 | 目标数量 | 建议原始采集量 | 主要用途 |
|---|---|---:|---:|---|
| P0 | 微博 | 清洗后不少于 900 条 | 1300-1800 条 | 态度倾向、关键词热度、情感分析、趋势分析 |
| P1 | 小红书 | 清洗后 300-500 条 | 500-800 条 | 真实使用场景、经验分享、评论反馈 |
| P1 | 知乎 | 清洗后 150-300 条 | 250-500 条 | 深度观点、伦理争议、风险解释 |
| P2 | 微信公众号 | 50-120 篇 | 50-120 篇 | 背景综述、媒体观点、案例支撑 |
| P2 | B站、贴吧、豆瓣 | 备用补充 | 按需要采集 | 话题补充，不作为主流程 |

### 4.2 微博采集方案

微博是本项目主数据源，必须优先完成。

目标：

- 原始采集 1300-1800 条。
- 清洗后不少于 900 条有效微博。
- 每个关键词尽量保留采集数量、时间范围、采集日期，保证可复现。

采集内容：

- 微博正文。
- 发布时间。
- 点赞数。
- 评论数。
- 转发数。
- 原始链接。
- 关键词来源。
- 是否转发。

采集注意事项：

- 只采集公开可见内容。
- 控制请求频率，避免对平台造成过大压力。
- 不采集私信、非公开聊天记录或需要绕过权限的数据。
- 作者信息只保存匿名哈希，不保存真实用户名。

评论抓取策略：

- 第一阶段先抓微博正文，目标是清洗后不少于 900 条有效微博。
- 第二阶段从有效微博中筛选高相关、高互动的 100-150 条微博，抓每条前 20-50 条评论。
- 微博评论主要用于观察公众态度和争议点，例如支持 AI 回消息、反对代聊、提醒隐私风险、调侃 AI 味等。
- 不对所有微博无限抓评论，避免成本过高和噪声过大。

### 4.3 小红书采集方案

小红书适合作为真实使用场景数据源。

目标：

- 清洗后 300-500 条有效文本。
- 包括笔记标题、正文、评论。

采集重点：

- “AI 帮我回消息”
- “AI 恋爱军师”
- “AI 高情商回复”
- “AI 分析聊天记录”
- “赛博军师”
- “AI 嘴替”
- “去 AI 味回复”
- “怎么拒绝别人”
- “怎么给老师发消息”
- “AI 味太重”
- “AI 代聊翻车”

采集注意事项：

- 优先采集文本内容，不采集图片中的完整聊天截图。
- 如果内容涉及具体个人身份、具体聊天对象，应在清洗时匿名化。
- 评论区可以采集，但要控制数量，优先选择高相关评论。

评论抓取策略：

- 小红书评论区价值最高，正式采集时每篇高相关笔记建议抓前 30-80 条评论。
- 优先保留高赞评论、楼主回复、具体经验评论和继续求助评论。
- 小红书评论重点分析真实经验、求助跟问和方法建议。

### 4.4 知乎采集方案

知乎适合补充“为什么”和“怎么看”。

目标：

- 清洗后 150-300 条有效回答或评论。

采集方法：

- 先通过关键词或搜索引擎找到 10-20 个相关问题。
- 再采集问题下高相关回答。
- 优先保留讨论 AI 恋爱咨询、聊天记录分析、代聊伦理、隐私风险和社交能力退化的回答。
- 回答本身可视作主文本，回答评论作为补充材料。
- 优先抓高赞回答及其评论，不对低相关问题大量抓评论。

### 4.5 微信公众号采集方案

微信公众号不作为大规模用户评论来源，只作为背景材料。

目标：

- 50-120 篇相关文章标题、摘要、正文片段。

采集方式：

- 半自动采集。
- 使用搜狗微信、微信搜索或浏览器搜索查找文章。
- 保存标题、发布时间、公众号名匿名化或规范化名称、摘要、正文片段、链接。
- 公众号只整理精选留言，不进入核心统计样本。

## 6. 数据字段设计

本项目采用“双层数据结构”：帖子表和评论表。

帖子表保存原帖、笔记、回答或文章的主体内容；评论表保存评论文本及其与原帖的关联关系。分析时不能让评论脱离原帖，因此评论必须保存 `parent_post_id`。

### 6.1 帖子表字段

帖子表至少保存以下字段：

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| post_id | string | 是 | 原帖唯一 ID |
| data_id | string | 是 | 唯一编号，建议由平台、日期和序号组成 |
| platform | string | 是 | 微博/小红书/知乎/微信公众号 |
| keyword | string | 是 | 采集关键词 |
| url | string | 是 | 原始链接 |
| publish_time | datetime/string | 否 | 发布时间 |
| crawl_time | datetime/string | 是 | 采集时间 |
| author_id_hash | string | 否 | 作者匿名编号，不保存真实用户名 |
| title | string | 否 | 标题，微博可为空 |
| text_raw | string | 是 | 原始文本 |
| text_clean | string | 是 | 清洗后文本 |
| like_count | int | 否 | 点赞数 |
| comment_count | int | 否 | 评论数 |
| repost_or_share_count | int | 否 | 转发或分享数 |
| collect_count | int | 否 | 收藏数，小红书可用 |
| source_type | string | 是 | 帖子/评论/回答/文章 |
| is_repost | bool/int | 否 | 是否转发 |

## 7. 数据清洗规则

### 7.1 删除规则

删除以下数据：

- 商业客服、销售话术、私域运营内容。
- AI 软件广告、课程推广、引流贴。
- 直播带货、企业微信、营销脚本相关内容。
- 无有效观点的转发。
- 只有表情、链接、话题标签的内容。
- 与人际沟通无关的 AI 新闻。
- 重复文本。
- 清洗后少于 10 个中文字符的文本。
- 明显机器生成、无真实讨论意义的垃圾内容。

删除或不纳入分析时必须填写 `delete_reason`，建议取值如下：

| delete_reason | 含义 |
|---|---|
| ad | 广告、营销、课程推广、引流 |
| irrelevant | 与 AI 辅助人际沟通无关 |
| duplicate | 重复文本 |
| too_short | 文本过短，清洗后少于 10 个中文字符 |
| no_opinion | 无有效观点，仅转发、链接、表情或话题 |
| privacy_risk | 包含过多个人隐私或完整聊天记录，不适合展示和分析 |
| platform_noise | 平台噪声或采集异常内容 |

### 7.2 保留规则

保留以下数据：

- 真实使用 AI 回消息的经历。
- 对 AI 回消息、AI 恋爱军师、AI 高情商回复的评价。
- 对 AI 分析聊天记录的支持或反对。
- 对 AI 味、隐私、代聊、依赖的担忧。
- 与大学生、年轻人、校园关系、恋爱关系、朋友关系、室友关系相关的文本。
- 玩梗和调侃内容，只要与主题相关，也应保留。

### 7.3 评论区清洗规则

评论区单独清洗，不完全沿用帖子规则。

- 原帖清洗后少于 10 个中文字符可以删除。
- 评论可以放宽到 5-8 个中文字符，因为短评论也可能有明确态度价值，例如“太像 AI 了”“这算代聊吧”“救命真好用”。
- 广告、引流、卖课、无关玩梗仍然删除。
- 涉及完整聊天记录、手机号、学校、宿舍号、真实姓名的评论，要脱敏或不展示。
- 如果评论只是表情、无意义附和或纯标点，标记为 `delete_reason = no_opinion`。
- 如果评论是评论区广告或工具推广，标记为 `delete_reason = ad`。
- 如果评论与原帖主题无关，标记为 `delete_reason = irrelevant`。

### 7.4 文本清洗步骤

1. 读取原始 CSV。
2. 统一字段名称。
3. 去除 HTML 标签。
4. 去除 URL。
5. 去除多余空格、换行、制表符。
6. 去除无意义符号和重复标点。
7. 对用户名、手机号、具体聊天对象等敏感信息进行匿名化。
8. 根据排除词标记疑似广告。
9. 根据关键词命中情况计算相关度评分。
10. 标注 `student_relevance_label`，区分 high/medium/low/unclear。
11. 删除重复文本，并写入 `delete_reason = duplicate`。
12. 删除过短文本，并写入 `delete_reason = too_short`。
13. 对不纳入分析的数据保留一份删除记录，便于解释样本筛选过程。
14. 输出清洗结果。

### 7.5 相关度评分建议

可采用简单可解释的规则评分：

| 命中内容 | 加分 |
|---|---:|
| 命中 AI 主体词 | +1 |
| 命中沟通动作词 | +2 |
| 命中人际关系词 | +2 |
| 命中情绪或风险词 | +1 |
| 命中商业排除词 | -3 |
| 文本长度低于 10 个中文字符 | -5 |

建议保留：

```text
relevance_score >= 2
```

低于阈值的数据可先放入待人工检查集合。

## 8. 标签体系设计

标签字段不全部设计成单选。一条文本可能同时属于多个场景、多个风险和多个主题。例如：

```text
把聊天记录发给 AI 分析 crush，感觉挺有用，但又怕隐私泄露。
```

这条文本至少同时命中：

- S2 恋爱关系咨询
- S3 关系判断分析
- U5 上传聊天记录
- K3 隐私泄露
- A2 谨慎接受

因此本项目采用“主标签 + 多标签”的两层结构：

- `primary_scene_label` 表示主场景，用于主要分布图。
- `scene_labels` 表示所有命中的场景标签，用于多标签统计。
- `primary_risk_label` 表示主要风险，用于主要风险分布图。
- `risk_labels` 表示所有命中的风险标签，用于交叉分析。
- `primary_topic_label` 表示主主题，用于主题主图。
- `topic_labels` 表示所有命中的主题标签，用于补充分析。
- `label_confidence` 表示标签置信度，便于后续人工复核优先检查低置信度样本。

## 9. 标签编码规则

### 9.1 编码方法

采用“机器规则初筛 + 人工抽样复核”的方式。

流程：

1. 根据 `label_dict.yaml` 中的关键词规则自动打标签。
2. 同时生成多标签字段和主标签字段。
3. 根据命中词数量、标签冲突情况、文本长度和平台噪声生成 `label_confidence`。
4. 每个平台随机抽取 10% 数据进行人工复核。
5. 优先复核 `label_confidence = low` 或 `student_relevance_label = unclear` 的样本。
6. 记录误判类型。
7. 修改标签词典。
8. 重新批量打标签。
9. 对高互动文本和典型文本进行额外人工检查。

### 9.2 规则示例

场景标签：

## 10. 文本分析计划

### 10.1 基础统计

需要完成以下统计：

- 各平台原始数据量。
- 各平台清洗后有效数据量。
- 各平台有效率。
- 各关键词命中数量。
- 不同 source_type 的数量。
- 发布时间分布。
- 点赞、评论、转发、收藏的分布。
- 高互动文本 Top 10%。

输出文件：


### 10.2 词频分析

1. 读取 `text_clean`。
2. 使用 jieba 分词。
3. 加载 `stopwords.txt`。
4. 加载 `keepwords.txt`，确保核心词不被错误切分或过滤。
5. 统计词频。
6. 输出 Top 30 高频词。
7. 生成词云图。

### 10.3 情感分析

情感标签不等于态度标签。

建议方法：

- 先使用 SnowNLP 或规则词典得到初始情感倾向。
- 再结合主题和语气进行修正。
- 对玩梗、反讽、复杂表达进行人工抽样检查。

分类：

- E1 正向：好用、救命、太会说、缓解焦虑、学到了。
- E2 中性：客观描述、教程、新闻、无明显情绪。
- E3 负向：恶心、可怕、不真诚、离谱、担忧、反感。
- E4 复杂混合：既觉得有用，又觉得危险或尴尬。

### 10.4 交叉分析

重点完成以下交叉表：

| 分析对象 | 目的 |
|---|---|
| primary_scene_label × primary_risk_label | 分析不同主使用场景对应的主要风险 |
| primary_scene_label × attitude_label | 比较不同主场景下用户态度差异 |
| motivation_labels × usage_labels | 分析不同动机导致的使用方式，按多标签展开统计 |
| platform × primary_topic_label | 比较不同平台的主话题差异 |
| platform × attitude_label | 比较不同平台的态度差异 |
| primary_topic_label × comment_attitude_to_post | 分析不同原帖主题下评论区态度 |
| primary_topic_label × comment_role_label | 分析不同主题下评论区的作用结构 |
| post_comment_consistency × primary_topic_label | 分析哪些主题更容易引发评论分化 |

## 13. 质量控制与科学性保证

### 13.1 数据质量控制

- 保留原始数据文件，清洗不覆盖原始文件。
- 每个平台保留采集关键词、采集时间和原始链接。
- 清洗规则写入脚本，保证可复现。
- 删除数据时保留删除原因或可复查标记。

### 13.2 标签质量控制

- 使用统一标签体系，不随意新增含义模糊的标签。
- 每个平台随机抽取 10% 数据进行人工复核。
- 对高互动文本额外复核。
- 记录误判类型并修正规则词典。
- 最终报告说明标签方法和局限性。

### 13.3 可视化质量控制

- 图表标题明确。
- 坐标轴和图例完整。
- 分类标签使用中文全称。
- 图表颜色保持一致。
- 不使用过多 3D 效果或装饰性图形。
- 词云只作为辅助图，不作为核心结论依据。
- 核心结论优先来自柱状图、热力图、交叉表和趋势图。

### 13.4 伦理与合规边界

- 只采集公开可见内容。
- 不采集私信、非公开聊天记录、付费内容或需要绕过权限的数据。
- 不保存真实用户名、手机号、学校、宿舍、具体聊天对象等个人身份信息。
- 所有案例必须匿名化。
- 不展示完整聊天截图。
- 控制采集频率，避免对平台造成负担。
- 数据仅用于课程实习和研究分析。

## 19. 风险与应对

| 风险 | 影响 | 应对方式 |
|---|---|---|
| 微博采集数量不足 | 无法满足 900 条有效样本要求 | 增加关键词批次，延长时间范围，优先采集高相关关键词 |
| 小红书登录态或反爬限制 | 数据补充不足 | 降低自动化依赖，改为半自动采集部分高相关笔记和评论 |
| 知乎关键词搜索噪声大 | 回答相关度不足 | 先人工筛选相关问题，再采集问题下回答 |
| 广告和营销内容较多 | 干扰分析结论 | 使用排除词和人工抽样检查 |
| 规则标签误判 | 降低分析可信度 | 每个平台抽样 10% 复核，修正规则后重跑 |
| 情感分析不准确 | 误读调侃和反讽文本 | 情感结果只作辅助，关键结论以主题和态度标签为主 |
| 图表过多但结论分散 | 报告缺乏主线 | 围绕 4 个研究问题组织图表和文字 |

## 20. 结论性执行原则

本项目不追求“爬得越多越好”，而追求数据与研究问题高度相关、清洗规则可解释、标签体系可复核、图表结论能支撑报告。

执行时应坚持以下原则：

- 先保证微博主样本质量，再扩展其他平台。
- 先完成 P0 图表，再考虑复杂可视化。
- 词云只能作为辅助，核心结论来自统计图、交叉表和热力图。
- 所有案例必须匿名化。
- 所有清洗、标签、统计过程尽量脚本化和 notebook 化，保证可复现。
